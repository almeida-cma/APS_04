O que é Análise de Requisitos?

Etapa fundamental para entender, documentar e validar necessidades de usuários e stakeholders. Define o que o sistema deve fazer e como se comportar.

📌 Exemplo prático: Em um e-commerce, a análise de requisitos identifica a necessidade de "carrinho de compras", "login seguro" e "tempo de resposta <2s".
⚙️

Requisitos Funcionais

Funcionalidades específicas: ações que o sistema deve executar.

✅ Exemplo: “O sistema deve permitir que o usuário recupere a senha por e-mail.”
🛡️

Requisitos Não Funcionais

Atributos de qualidade: desempenho, segurança, usabilidade, disponibilidade.

✅ Exemplo: “O sistema deve suportar 500 usuários simultâneos com tempo de resposta < 1.5s.”
🧑‍💼

Requisitos de Usuário x Sistema

Usuário: necessidades dos usuários finais. Sistema: especificações técnicas.

Exemplo Usuário: “Visualizar histórico de pedidos.”
Exemplo Sistema: “Banco de dados PostgreSQL com índice em user_id.”

📊 Tipos de Requisitos em ação

Cenário: Aplicativo de entregas. Identifique qual tipo de requisito se encaixa:

  • 🔹 "O app deve enviar notificação push ao entregador" → Funcional
  • 🔹 "O aplicativo deve usar criptografia de ponta a ponta" → Não Funcional (Segurança)
  • 🔹 "O sistema deve estar disponível 99,9% do tempo" → Não Funcional (Disponibilidade)
🗣️

Técnicas de Elicitação

  • Entrevistas com stakeholders
  • Questionários e pesquisas
  • Workshops e brainstorming
  • Observação direta e etnografia
  • Prototipação interativa
📐

Modelagem Visual (UML)

Diagramas de Caso de Uso, sequência, atividades – ajudam a visualizar interações.

📘 Exemplo - Caso de Uso: Ator "Cliente" → caso "Realizar Pedido" e "Acompanhar Entrega".
📄

Documentação de Requisitos

Especificação textual, matriz de rastreabilidade, protótipos de baixa fidelidade.

Ferramentas: Jira, Confluence, IBM DOORS, Lucidchart.
📌 Exemplo de artefato: Histórias de Usuário (Ágil)
"Como usuário comum, eu quero filtrar produtos por preço para encontrar ofertas rapidamente."
Critérios de aceitação: slider de preço, atualização em tempo real.

💧 Modelo Cascata

Fases sequenciais: Requisitos → Design → Implementação → Testes → Manutenção. Cada fase termina antes da próxima.

✅ Ideal para projetos com requisitos estáveis e bem definidos (ex: sistemas governamentais, controle de tráfego aéreo).
📌 Exemplo prático: Software para folha de pagamento – regras fiscais fixas, pouca mudança.

🔄 Modelos Ágeis (Scrum/Kanban)

Iterativo, incremental, com entregas contínuas e adaptação aos feedbacks dos stakeholders.

✅ Ideal para projetos com requisitos dinâmicos (ex: aplicativos de marketplace, startups).
📌 Exemplo: Time Scrum trabalha em Sprints de 2 semanas, refinando backlog com Product Owner.

📌 Quando aplicar cada um?

Cascata: Requisitos claros, pouca mudança esperada, contratos rígidos.
Ágil: Necessidade de entregas rápidas, incertezas, envolvimento constante do cliente.

🔍

Rastreabilidade de Requisitos

Conexão entre requisitos, casos de uso, testes e implementação. Permite verificar se tudo foi atendido.

Matriz de Rastreabilidade: Requisito R001 → Caso de teste CT-12 → Módulo de autenticação.
🔄

Gerenciamento de Mudanças

Processo controlado para avaliar impacto, aprovar e implementar alterações nos requisitos.

✅ Fluxo: Solicitação → Análise de impacto → Aprovação (CCB) → Atualização da documentação.
✔️

Validação x Verificação

Validação: "construímos o produto certo?" (stakeholders aprovam).
Verificação: "construímos corretamente?" (técnica).

📌 Exemplo de Rastreabilidade (Cenário)

Projeto: Sistema de Reservas de Hotel

  • RF-01: Reservar quarto → Implementado no módulo Reserva → Teste TR-08 validado.
  • RNF-04: Sistema deve responder em até 1s → Teste de carga aprovado.

🎯 Por que priorizar requisitos?

Evita escopo descontrolado, foca no MVP (Produto Mínimo Viável) e otimiza recursos. Métodos como MoSCoW, Story Mapping e Análise de Valor são essenciais.
